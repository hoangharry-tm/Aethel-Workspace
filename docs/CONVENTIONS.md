# Docs Folder Conventions

This file is the authoritative reference for what goes in `docs/`, how files are named, and how diagrams are maintained. Read it before adding any file to this directory.

---

## What belongs in `docs/`

The `docs/` folder holds human-authored material that captures **design rationale, architectural decisions, operational guides, and project planning**. It is not a build artifact directory.

Appropriate content:

- Architecture decision records and design rationale (why a pattern was chosen, not just what it is)
- Mermaid diagram sources for system-level visualizations
- Developer guides explaining non-obvious subsystems (blueprint loading, migration rendering, query registry)
- IT admin guides for configuring and customising the system
- Agile sprint plans and implementation roadmaps

---

## What does NOT belong in `docs/`

- Auto-generated API reference documentation (OpenAPI/Swagger output, godoc HTML)
- Build artifacts (compiled binaries, coverage reports, test output)
- Log files or runtime output of any kind
- Database dumps or data exports
- Anything generated by a tool that can be regenerated from source

If a tool generates it, it does not live here.

---

## Naming conventions

### Architecture documents

Prefix: `architecture-`

```
architecture-code.md        # Package structure and code design patterns
architecture-server.md      # Server components, middleware stack, request flow
architecture-api-routes.md  # API route design and blueprint configuration
architecture-security.md    # Embedded security measures
```

### Diagrams

Directory: `docs/diagrams/`
Extension: `.mmd` (Mermaid source)

```
diagrams/architecture-overview.mmd     # High-level system overview
diagrams/architecture-code.mmd         # Package dependency graph
diagrams/server-request-flow.mmd       # HTTP request lifecycle
diagrams/api-route-resolution.mmd      # Route resolution at startup
```

Diagrams are always Mermaid source files. They are referenced from Markdown documents using a relative path. They are never committed as rendered images.

### Guides

Suffix: `-guide.md`

```
go-developer-guide.md           # Reference for Go engineers working in aethel-core/
it-customization-guide.md       # Reference for IT administrators deploying the system
server-blueprint-conventions.md # YAML blueprint field reference
```

### Plans

Suffix: `-plan.md`

```
agile-implementation-plan.md    # Sprint breakdown for Phase 2 Go backend
```

### Strategy and design documents

Suffix: none (free-form descriptive name)

```
migration-strategy.md           # Rationale for blueprint-rendered SQL migrations
```

---

## Required header format

Every Markdown file in `docs/` must open with:

```markdown
# <Title>

**Audience:** <who should read this>
**Status:** <Draft | Active | Superseded>
```

The `Audience` line names the reader: `Go engineers`, `IT administrators`, `project leads`, `all contributors`. The `Status` line indicates whether the document reflects the current state of the system.

Diagram files (`.mmd`) do not require a header — Mermaid files contain only diagram syntax.

---

## How to add a new diagram

1. Create a `.mmd` file in `docs/diagrams/` using the naming convention above.
2. Write valid Mermaid syntax. Validate it at [mermaid.live](https://mermaid.live) or in the VS Code Mermaid Preview extension before committing.
3. Reference the diagram from the relevant Markdown document using a relative path and a plain link (not an embedded image, because GitHub renders `.mmd` files natively):

```markdown
Diagram: [docs/diagrams/architecture-code.mmd](diagrams/architecture-code.mmd)
```

4. If the diagram is large or complex, break it into multiple smaller diagrams rather than adding a single sprawling one.

Do not commit rendered PNG or SVG exports. The `.mmd` source is the single source of truth.
